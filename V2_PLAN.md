# CookiePilot V2 — แผนอัพเกรด "Humanization"

> เป้าหมาย: ทำให้บอทที่รันวันละ 16–20 ชม. ดูเป็น "คนเล่นจริงที่เล่นหนัก" ไม่ใช่มาโครที่กดจังหวะเดิมซ้ำพันรอบ
> เงื่อนไขตายตัว: **ห้ามแตะ logic รีโรล Double Coins** (multibuy_until_target) — คนเก็บเหรียญจริงก็รีโรลหาอันนี้อยู่แล้ว พฤติกรรมนี้ไม่ผิดปกติ

---

## Phase 1 — Tier S: Session Planner (แบ่งช่วงพักตอนจบเกม)

**ปัญหา:** ตอนนี้บอทเล่นต่อเนื่องไม่หยุดเลยตลอด 16–20 ชม. จบเกม → รีโรล → เล่น → จบเกม วนทันทีทุกรอบ ไม่มีมนุษย์คนไหนทำแบบนี้ได้

**แนวคิด (ปรับตาม feedback):** เพิ่มชั้น "SessionPlanner" ที่แทรกช่วงพัก **เฉพาะที่รอยต่อจบเกม** (จุด `RESULT → REROLL` ใน `bot.run()` ตรงที่ `loops_done` ถูกนับ) เกมจะเห็นว่า "ผู้เล่นเล่นจบรอบแล้ววางมือถือแป๊บนึง" ไม่มีการหยุดกลางด่าน

- **พักหลังจบเกมทุกรอบ: สุ่ม uniform 0–60 วินาที** (ตัดพักยาว Meso/Macro ออกตาม feedback — ผู้ใช้จัดการเวลาเปิด-ปิดเองอยู่แล้ว)
- ตอนพัก: บอทกด OK กลับล็อบบี้ตามปกติแล้ว "นิ่งสนิท" — ไม่ยิง input ใด ๆ ใช้ sleep แบบขัดจังหวะได้ (เช็ค `self.stop` ทุก 0.5 วิ) กดหยุดบอทได้ทันที
- **Fatigue drift:** ยิ่งเล่นต่อเนื่องนานในเซสชัน `HUMAN_GAP_MU` ค่อย ๆ เพิ่ม (กดช้าลงนิด ๆ จนสุดที่ +0.20 เมื่อเล่นครบ 6 ชม.) — รีเซ็ตเมื่อกดเริ่มบอทใหม่
- **Parameter reroll:** ทุก 60–150 นาที (สุ่ม) สุ่มค่าจังหวะกดชุดใหม่ในกรอบ (MU ±0.15, SIGMA ±0.08, BURST_CHANCE ±0.05, IDLE_CHANCE ±0.03) → สถิติแต่ละช่วงของวันไม่เหมือนกัน เหมือนอารมณ์/ท่านั่งคนเปลี่ยน (ทับเฉพาะในหน่วยความจำ ไม่เขียนลง settings.json)

**config ใหม่ (override ผ่าน settings.json ได้ทั้งหมด):**
```
SESSION_PLANNER        = True
BREAK_AFTER_GAME_RANGE = (0, 60)     # วินาที — พักหลังจบเกมทุกรอบ
FATIGUE_ENABLED        = True
FATIGUE_FULL_HOURS     = 6.0
FATIGUE_MU_MAX_ADD     = 0.20
SESSION_REROLL_PARAMS  = True
PARAM_REROLL_EVERY_MIN = (60, 150)   # นาที
```

**ต้นทุน:** พักเฉลี่ย ~30 วิ/เกม — เกมละ ~4-5 นาทีเสียเวลาราว 10% แลกกับ cadence ที่ไม่เท่ากันเป๊ะทั้งวัน วัดผลจากระยะห่าง timestamp ใน `coin_logs/coins.csv`

**สถานะ: ✅ ทำแล้ว** — โมดูลใหม่ `humanizer.py` (คลาส SessionPlanner) + hook 1 จุดใน `bot.run()` — ไม่แตะ state machine เดิม

## Phase 2 — Tier A: พฤติกรรมในด่าน (สไลด์ / พลาดบ้าง / จุดกดกว้างขึ้น)

### 2.1 เปิดสไลด์ (ตอนนี้ HUMAN_SLIDE_CHANCE = 0.0 → ไม่เคยสไลด์เลย)
- เริ่มที่ `HUMAN_SLIDE_CHANCE = 0.06` (ประมาณ 1 ใน 17 จังหวะ)
- **ระยะกดค้างต้องสุ่ม** — ตอนนี้ `SLIDE_HOLD_SEC = 0.35` ตายตัวทุกครั้ง เป็นลายเซ็นหุ่นชัด ๆ → เปลี่ยนเป็นช่วง `(0.25, 0.60)` และนาน ๆ ทีสไลด์ยาว `(0.8, 1.2)` โอกาส ~15% ของการสไลด์
- **คอมโบ jump→slide:** คนเล่น Cookie Run จริงชอบกดกระโดดแล้วสไลด์ต่อทันที → เพิ่ม `HUMAN_COMBO_CHANCE = 0.04` (กระโดด → เว้น 0.15–0.35 วิ → สไลด์)
- ความเสี่ยง: สไลด์ผิดจังหวะ = ตกหลุม/ชนได้ → **วัดผลจาก coins.csv ก่อน/หลังเปิด** ถ้าเหรียญเฉลี่ยต่อรอบตกเกิน ~10% ให้ลด chance ลงครึ่งหนึ่ง

### 2.2 จงใจพลาดบ้าง (intentional miss)
คนจริงพลาด 3 แบบ — ใส่ทั้งสามแบบด้วยโอกาสต่ำ:
- **กดหลุดปุ่ม (fumble):** โอกาส ~1.5% แตะเยื้องไกล 50–120px จากปุ่ม (นิ้วลื่น) — เกมไม่รับ input นั้น = เสียจังหวะไปเฉย ๆ เหมือนคนพลาดจริง
- **react ช้า:** โอกาส ~3% หน่วง gap ยาวพิเศษ 1.5–2.5 วิ (เกิน HUMAN_GAP_MAX ปกติ) — เหมือนเผลอ/หลุดโฟกัส
- **นินจา relay ไม่กดทันที:** ตอนนี้บอทกด relay ภายใน 0.5 วิทุกครั้งเป๊ะ (bot.py:469-473) → เพิ่ม reaction delay สุ่ม 0.25–0.9 วิ ก่อนกด และโอกาส ~5% "มองไม่เห็นรอบแรก" (ข้าม 1 cycle แล้วค่อยกด)

### 2.3 ขยายการแกว่งจุดกด
- `JUMP_JITTER: 28 → 36` (gaussian σ 14 → 18)
- Drift แรงขึ้น: `HUMAN_DRIFT_STEP: 3.0 → 4.5`, `HUMAN_DRIFT_MAX: 18 → 32`
- **Anchor bias ต่อเซสชัน:** สุ่ม offset ฐานนิ้ว ±(15–25)px หนึ่งครั้งต่อเซสชัน (ผูกกับ Phase 1 parameter reroll) — คนแต่ละช่วงวันวางนิ้วไม่ตรงเดิม ทำให้ heatmap จุดกดรายวันไม่เป็นวงกลมสมมาตรจุดเดียว

**config ใหม่:**
```
HUMAN_SLIDE_CHANCE      = 0.06
HUMAN_SLIDE_HOLD        = (0.25, 0.60)
HUMAN_SLIDE_LONG_CHANCE = 0.15
HUMAN_SLIDE_LONG_RANGE  = (0.8, 1.2)
HUMAN_COMBO_CHANCE      = 0.04
HUMAN_COMBO_GAP         = (0.15, 0.35)
HUMAN_FUMBLE_CHANCE     = 0.015
HUMAN_FUMBLE_DIST       = (50, 120)
HUMAN_LAPSE_CHANCE      = 0.03
HUMAN_LAPSE_RANGE       = (1.5, 2.5)
RELAY_REACT_RANGE       = (0.25, 0.9)
RELAY_MISS_CHANCE       = 0.05
SESSION_ANCHOR_BIAS     = (15, 25)
```

**สถานะ: ✅ ทำแล้ว** — `one_cycle()`/relay ใน bot.py, `slide(hold_sec=)` ใน adb.py, anchor bias ใน humanizer.py (สุ่มตอนเริ่ม + ทุก reroll)

## Phase 3 — Tier B คืออะไร: จังหวะเมนู/การนำทาง (นอกด่าน)

**อธิบาย:** Tier A คือพฤติกรรม "ตอนวิ่งในด่าน" — ส่วน Tier B คือทุกอย่าง **นอกด่าน**: กดเมนู รีโรล ติ๊กบูสต์ กด OK หน้าผลลัพธ์ ตรงนี้ตอนนี้ **เป็นหุ่นยนต์กว่าการกระโดดมาก** เพราะ:

1. **ดีเลย์เมนูตายตัวเป๊ะทุกรอบ:** กดกล่อง → รอ 0.8 วิ → กด Multi → รอ 1.0 วิ → ติ๊ก → รอ 0.4 วิ ... ค่าเดิมเป๊ะระดับมิลลิวินาทีซ้ำวันละหลายร้อยรอบ (bot.py:237, 240, 260 ฯลฯ) นี่คือ fingerprint มาโครที่ชัดที่สุดรองจากการเล่นไม่พัก
2. **กดปุ่มทันทีที่เห็น:** เจอหน้า Result ปุ๊บกด OK ปั๊บ — คนจริง **หยุดดูคะแนน/เหรียญ 1.5–4 วิ** ก่อนกดเสมอ
3. **จุดกดเมนูเยื้องแค่ ±7px:** จุดเดิมแทบเป๊ะตลอดชีวิต

**แผน:**
- สร้าง helper `human_sleep(base)` — คูณ base ด้วยตัวสุ่ม log-normal (×0.7–×1.8) แล้วไล่แทน `time.sleep()` คงที่ทุกตัวใน flow เมนู (`ensure_on_boost_screen`, `multibuy_until_target`, `ensure_boosts_selected`, `state_result`)
- **Reading pause หน้า Result:** ก่อนกด OK หน่วง 1.5–4 วิ (นาน ๆ ทีดูนาน 5–8 วิ) — สมเหตุสมผลมากเพราะบอทนี้เก็บเหรียญ คนจริงต้องดูตัวเลข
- หน่วงก่อนกด Play เริ่มเกม 0.5–1.5 วิ (คนขยับนิ้วไปหาปุ่ม)
- jitter เมนู: `TAP_JITTER 7 → 10` แบบ gaussian **ยกเว้น** จุดเสี่ยง (ติ๊กบูสต์ / MULTI_SELECT_TARGETS ที่กดพลาดแล้ว toggle ผิด) — เพิ่ม param `jitter=` รายจุดให้จุดเสี่ยงคง 5–7px

**ข้อควรระวัง:** อย่าไปแตะ timeout/การรอ template (MULTIBUY_TIMEOUT, FAST_START_*) — พวกนั้นเป็นการ "รอเกม" ไม่ใช่ "จังหวะนิ้วคน" ปรับแล้วมีแต่เสียเสถียรภาพ

**สถานะ: ✅ ทำแล้ว** — `humanizer.menu_pause()` (ยืดจาก base เสมอ ไม่หด — base จูนกับแอนิเมชันเกมไว้แล้ว), `_msleep`/`_pause` ใน bot.py แทน sleep คงที่ทุกจุดใน flow เมนู, reading pause ก่อนกด OK (config: `MENU_HUMAN_DELAY`, `PLAY_PRESS_DELAY`, `READ_RESULT_*`), `TAP_JITTER 7→10` + `SAFE_TAP_JITTER=6` สำหรับจุดเสี่ยง toggle

## Phase 4 — Tier C คืออะไร: ลายเซ็นระดับ input/อุปกรณ์ (ลึกสุด, ทำน้อยสุด)

**อธิบาย:** Tier C คือสิ่งที่เกมจะจับได้ก็ต่อเมื่อวิเคราะห์ **touch event ดิบ** ระดับ kernel — ลึกกว่าจังหวะกด:

1. **แตะแบบนิ้วไม่ขยับเลย:** ตอนนี้ tap-hold ทำด้วย `swipe x y x y` (จุดเดิมเป๊ะ — adb.py:204) และสไลด์คือกดค้างนิ่งสนิท จุดเดียว 100% — นิ้วคนจริงขยับ 1–5px ระหว่างสัมผัสเสมอ **แก้ถูกมาก:** swipe ไปจุดข้าง ๆ (สุ่ม 1–4px) แทนจุดเดิม และสไลด์ให้ลากนิ้วเลื่อน 8–20px ระหว่างกดค้าง — แก้ ~5 บรรทัดใน `adb.py` (`tap`, `hold`, `slide`)
2. **ทุก touch มาจาก `adb shell input`:** event source เป็น virtual device ของ ADB — เกมฝั่ง client ที่ตรวจ input device จับได้ในทางทฤษฎี แต่แก้ไม่ได้ถ้าไม่เขียน injector ระดับ `sendevent` เอง (งานใหญ่ ผลตอบแทนต่ำ)
3. **LDPlayer เองก็คืออีมูเลเตอร์:** device fingerprint (รุ่นเครื่อง, sensor, build props) บอกอยู่แล้วว่าเป็น emulator — เกมนี้อนุญาตให้เล่นบน emulator ได้ ถ้าเขาจะแบน emulator เขาแบนได้เลยโดยไม่ต้องดูพฤติกรรม → **ไม่ต้องทำอะไร**
4. **การแคปจอ (screencap):** ทำงานฝั่ง host ผ่าน ADB ไม่มี process ในเครื่อง Android ที่เกมมองเห็น → **ไม่ใช่ความเสี่ยง ไม่ต้องแก้**

**สรุป Tier C:** ทำแค่ข้อ 1 (micro-movement ใน tap/hold/slide — ถูกและได้ผล) ข้อ 2–4 บันทึกไว้เฉย ๆ ว่ารับความเสี่ยงได้/ไม่ใช่ความเสี่ยง

**สถานะ: ✅ ทำแล้ว (ข้อ 1)** — `Adb._travel_end()`: แตะ/กดค้างมีนิ้วขยับ 1–4px, สไลด์ลากนิ้ว 8–20px ระหว่างกดค้าง (config: `HUMAN_TOUCH_TRAVEL`, `HUMAN_TAP_TRAVEL`, `HUMAN_SLIDE_TRAVEL`)

## โหมดเก็บกล่อง (box) — ครอบคลุมแล้ว

ของที่ใช้ร่วมกับโหมด coin โดยอัตโนมัติ: พักหลังจบเกม (Phase 1), จังหวะเมนู/ติ๊กบูสต์/หน่วงก่อนกด Play (Phase 3), ดูคะแนนก่อนกด OK (Phase 3), micro-movement (Phase 4)

ของเฉพาะโหมด box ที่เพิ่ม (**✅ ทำแล้ว**):
- หน่วง react 0.3–0.9 วิ ก่อนแตะไอคอน Fast Start และก่อนกดปุ่ม Buy (`BOX_REACT_RANGE`) — เดิมกดเฟรมแรกที่ template โผล่ (หน้าเตรียมตัวไม่มีจับเวลา หน่วงได้ไม่เสี่ยง)
- ปุ่ม "Tap to activate Fast Start" **ไม่หน่วง — กดทันทีเสมอ** (ปุ่มค้างแค่ ~3 วิ พลาดคือเสียบูสต์ทั้งด่าน) และช่วง 30 วิแรกเช็คจอถี่ขึ้นเป็นทุก 0.2 วิ ให้เจอปุ่มเร็วและกดทันทุกครั้ง — ความหน่วงธรรมชาติจาก screencap+จับภาพก็สุ่มพออยู่แล้ว ไม่เป๊ะระดับเฟรม
- รอ toast หลังซื้อเปลี่ยนเป็น `_msleep` (ยืดสุ่ม ไม่ใช่ 2.0 วิเป๊ะ)
- ส่วนตอนวิ่ง "ไม่กดอะไรเลยทั้งด่าน" คงไว้ตามสเปกโหมด — คนใช้ Fast Start ฟาร์มแบบ AFK ก็ทำแบบนี้จริง ไม่ใช่ลายเซ็นหุ่น

**สิ่งที่ไม่ทำใน V2:**
- ไม่แตะ Multi-Buy / รีโรล Double Coins ทุกกรณี
- ไม่ทำ sendevent injector / ปลอม device fingerprint
- ไม่ทำระบบตารางเวลานอน (ผู้ใช้เป็นคนเปิด–ปิดบอทเองอยู่แล้ว วันละ 16–20 ชม. — จุดเริ่ม/จบวันจึงสุ่มตามมือคนโดยธรรมชาติ)
